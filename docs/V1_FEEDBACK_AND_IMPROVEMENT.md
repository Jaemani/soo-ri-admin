## 2. 멘토링 피드백 분석

### 2.1 받은 피드백 요약

멘토링 및 심사 과정에서 받은 주요 피드백을 3가지 카테고리로 분류합니다.

#### 🔴 Critical (핵심 우려)

| 피드백 | 의미 | 심각도 |
|--------|------|--------|
| **"단순한 GPT Wrapper 아닌가?"** | AI 활용이 프롬프트 주입 수준에 그침 | 높음 |
| **"데이터가 쌓이면 어떤 가치가 있나?"** | 데이터 활용 전략 부재 | 높음 |
| **"사용자가 계속 쓸 이유가 뭔가?"** | 리텐션 전략 부재 | 높음 |

#### 🟡 Important (개선 필요)

| 피드백 | 의미 | 심각도 |
|--------|------|--------|
| **"복지 서비스 데이터가 늘어나면?"** | 확장성 한계 | 중간 |
| **"GPS 데이터로 뭘 더 할 수 있나?"** | IoT 데이터 활용 부족 | 중간 |
| **"보호자 연동은 어떻게?"** | 기존 `guardianIds` 필드 미활용 | 중간 |

#### 🟢 Nice-to-have (추가 제안)

| 피드백 | 의미 | 심각도 |
|--------|------|--------|
| **"지자체와 협업 가능성은?"** | B2G 비즈니스 모델 | 낮음 |
| **"다른 지역 확장은?"** | 스케일업 전략 | 낮음 |

### 2.2 피드백의 근본 원인

```
┌─────────────────────────────────────────────────────────────┐
│                    V1의 본질적 한계                          │
├─────────────────────────────────────────────────────────────┤
│  "정적 정보 제공 앱" (Static Information Provider)           │
│                                                             │
│  • 데이터 흐름: 단방향 (시스템 → 사용자)                      │
│  • AI 역할: 요약/설명 (Passive)                              │
│  • 사용자 행동: 열람만 가능 (Read-only)                       │
│  • 데이터 가치: 일회성 소비 (No Accumulation)                 │
└─────────────────────────────────────────────────────────────┘
```

## 3. V1의 기술적 한계

### 3.1 아키텍처 한계

#### 동기식 처리의 문제

```javascript
// 현재 V1 코드 (pipeline.js)
exports.runWelfarePipeline = async (userId) => {
  const services = loadServices();           // ~100ms
  const userContext = await buildUserContext(userId, db);  // ~500ms
  const candidates = selectCandidates(userContext, services);  // ~10ms
  const reportData = await generateReport(userContext, candidates);  // ~5-30초 ⚠️
  await saveReport(userId, reportData);      // ~200ms
  return { success: true, report: reportData };
};
```

**문제점**:
- GPT-5 응답이 30초 이상 걸리면 **Cloud Functions 타임아웃** (기본 60초)
- 사용자 증가 시 **동시 요청 처리 불가**
- 실패 시 **재시도 로직 없음**

#### 메모리 기반 데이터 처리의 한계

```javascript
// 현재 V1 코드 (csvLoader.js)
let cachedServices = null;  // 전역 캐싱

// 문제: 복지 서비스가 1000개 이상이면?
// - 메모리 부족
// - GPT 토큰 제한 초과
// - Cold Start 시간 증가
```

### 3.2 AI 활용 한계

#### Rule-based 필터링의 한계

```javascript
// 현재 V1 코드 (candidateFilter.js)
if (tags.disability && userContext.recipientType !== 'disabled') return false;
if (tags.lowIncome && userContext.recipientType !== 'lowIncome') return false;
```

**문제점**:
| 시나리오 | 현재 동작 | 기대 동작 |
|----------|-----------|-----------|
| "휠체어 타이어 수리" 검색 | 매칭 실패 (키워드 없음) | "보장구 수리 지원" 추천 |
| 새로운 복지 서비스 추가 | 태그 수동 추가 필요 | 자동 분류 |
| 사용자 맥락 변화 | 반영 불가 | 실시간 반영 |

#### LLM 활용이 "Wrapper" 수준

```javascript
// 현재: 단순 프롬프트 주입
const inputData = {
  user_context: { recipientType, hasSensorId, stats },
  candidate_services: candidateServices.slice(0, 10)
};
// GPT는 주어진 데이터를 "요약"할 뿐, "추론"하지 않음
```

### 3.3 데이터 활용 한계

#### GPS 데이터의 얕은 활용

```javascript
// 현재 V1: 단순 통계
return {
  totalDistance: 17.93,  // 합계
  trend: 'stable'        // 증감 추세
};

// 활용되지 않는 가치:
// - 급감속/급정거 좌표 → 위험 구간 탐지
// - 주행 패턴 변화 → 건강 이상 징후
// - 기상 데이터 연계 → 고장 예측
```

#### 피드백 루프 부재

```
현재: 사용자 → 리포트 열람 → 끝

필요: 사용자 → 리포트 열람 → 피드백 제공 → 다음 추천 개선
                                    ↑
                              RLHF 데이터
```

## 4. 개선 방향 도출

### 4.1 피드백별 개선 방향

| 피드백 | 근본 원인 | 개선 방향 |
|--------|-----------|-----------|
| "GPT Wrapper" | AI가 요약만 수행 | **RAG 도입**으로 검색+추론 분리 |
| "데이터 가치" | 일회성 소비 | **피드백 루프**로 데이터 축적 |
| "리텐션" | 일방향 정보 제공 | **행동 유도형** 알림 및 보호자 연동 |
| "확장성" | 동기식 처리 | **비동기 이벤트 기반** 아키텍처 |
| "GPS 활용" | 단순 통계 | **패턴 분석 및 위험 감지** |

### 4.2 핵심 전환 (Paradigm Shift)

```
┌─────────────────────────────────────────────────────────────┐
│                V1 → V2 패러다임 전환                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   V1: "정적 정보 제공" (Static Info Provider)               │
│        → "너에게 맞는 복지 혜택은 이거야"                    │
│                                                             │
│   V2: "동적 맥락 기반 행동 유도" (Actionable Intelligence)   │
│        → "너의 주행 패턴을 보니 배터리가 위험해,             │
│           그리고 이 도로가 위험하니 구청에 신고해 줄게"       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 4가지 핵심 축 (Core Pillars)

| Pillar | V1 | V2 | 비즈니스 가치 |
|--------|----|----|---------------|
| **Intelligence** | Rule-based + LLM Summary | Vector Search (RAG) + Personalization | "우리만의 데이터 검색 엔진" |
| **Data** | 주행 거리 (단순 통계) | 주행 패턴 분석 (위험 감지) | "B2G 도로 위험 정보 제공" |
| **Interaction** | 조회 (Read) | 피드백 + 보호자 공유 (Action) | "리텐션 및 바이럴 루프" |
| **Architecture** | On-demand (Sync) | Event-driven (Async) | "엔터프라이즈급 확장성" |

## 5. V2 진화의 필요성

### 5.1 기술 포트폴리오 관점

V2 구현 시 어필할 수 있는 기술적 깊이:

| 기술 | 포트폴리오 가치 |
|------|-----------------|
| **Cloud Tasks** | "대규모 비동기 처리 경험" |
| **Vector DB + RAG** | "LLM 애플리케이션 아키텍처 설계" |
| **Time-series 분석** | "IoT 데이터 파이프라인 구축" |
| **FCM + Guardian Mode** | "실시간 알림 시스템 구현" |

### 5.2 제품 가치 관점

| 관점 | V1 | V2 |
|------|----|----|
| **사용자 가치** | 복지 정보 열람 | 위험 예방 + 행동 유도 |
| **데이터 가치** | 일회성 소비 | 축적 → 인사이트 → 개선 |
| **비즈니스 모델** | B2C (정보 제공) | B2C + B2G (데이터 판매) |

### 5.3 경쟁 우위 관점

```
┌─────────────────────────────────────────────────────────────┐
│              V2가 만드는 차별화 포인트                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. "단순 GPT Wrapper가 아닌, 우리만의 검색 엔진 보유"        │
│     → Vector DB + 도메인 특화 임베딩                         │
│                                                             │
│  2. "지자체에 도로 위험 정보를 제공하는 B2G 모델"             │
│     → GPS 데이터 기반 Safety Map                            │
│                                                             │
│  3. "사용자 피드백으로 계속 똑똑해지는 시스템"                │
│     → RLHF 기반 추천 개선                                   │
│                                                             │
│  4. "보호자 연동으로 사회적 가치 창출"                        │
│     → Guardian Mode + 이상 징후 알림                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 📝 결론

V1은 **MVP로서 성공적**입니다. 하지만 멘토링 피드백이 지적한 것처럼:

1. **"GPT Wrapper"를 넘어서는 AI 활용**이 필요합니다.
2. **데이터가 쌓일수록 가치가 커지는 구조**가 필요합니다.
3. **사용자가 계속 돌아오는 이유**를 만들어야 합니다.

이를 위해 V2에서는:

- **RAG 기반 의미론적 검색**으로 AI 깊이 확보
- **비동기 이벤트 기반 아키텍처**로 확장성 확보
- **피드백 루프 + 보호자 모드**로 리텐션 확보
- **GPS 패턴 분석**으로 데이터 가치 창출

다음 문서 **[V2_IMPLEMENTATION_ROADMAP.md]**에서 구체적인 구현 계획을 다룹니다.
